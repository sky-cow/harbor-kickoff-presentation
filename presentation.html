Key API Fields Explained

transfer_purpose (Required)

Why the transfer is happening

  • "Salary", "Investment", "Payment for services", "Personal transfer"

Impact: Compliance reporting, AML risk assessment

is_self_transfer (Required)

Is sender = recipient?

  • true: Same person moving own funds
  • false: Sending to someone else

Impact: Different compliance checks

application_transfer_uuid (Required)

Your unique ID for this transfer

  • Use for reconciliation between systems
  • Enables idempotency (safe retries)
  • Critical for support requests

Best Practice: txn_${timestamp}_${randomID}

Payment Methods - Store vs. Dynamic

Option A: Pre-Store Payment Methods (Recommended)

Step 1: Create once

POST /customers/{uuid}/bank_accounts { "account_number": "123456789", "routing_number": "021000021", "account_holder_name": "John Doe", "bank_name": "Chase" }

Step 2: Reference in transfers

{ "destination": { "customer_bank_account_id": "bank_xyz123" } }

Benefits: Faster checkout, better UX, less data per call

Option B: Provide Details Per Transfer

{ "destination": { "account_number": "123456789", "routing_number": "021000021", "bank_name": "Chase", "account_holder_name": "John Doe" } }

Benefits: More flexible, no storage, good for one-time transfers
